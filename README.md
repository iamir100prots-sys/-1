# -1
ЛИМТЕР ВРЕМЕНИ
Вот структурированный план проекта **«Лимитер времени в соцсетях»** по заданным разделам:

---

### 1. ЦЕЛЬ ПРОЕКТА

**Основная цель:** Разработать инструмент, который помогает пользователям осознанно управлять временем, проводимым в социальных сетях и на развлекательных сайтах, снижая уровень цифровой зависимости и повышая личную продуктивность.

**Задачи:**
*   Предоставить пользователю прозрачные данные о его привычках (метрики).
*   Дать возможность устанавливать жесткие или гибкие ограничения.
*   Создать мотивационную среду для формирования здоровых цифровых привычек.

---

### 2. ОБЩАЯ АРХИТЕКТУРА

Для эффективной работы лимитера (особенно на разных платформах) архитектура должна включать несколько слоев:

*   **Клиентская часть (Фронтенд):**
    *   **Мобильное приложение** (нативное для Android/iOS или кроссплатформенное).
    *   **Браузерное расширение** (для Chrome, Firefox, Safari) — для отслеживания времени на компьютере.
*   **Ядро логики (Локальное):**
    *   **Трекер использования:** Модуль, который на уровне ОС или браузера собирает данные о запущенных приложениях и открытых вкладках.
    *   **Менеджер блокировки:** Компонент, который физически ограничивает доступ (закрывает приложение, подменяет контент сайта заглушкой) при срабатывании лимита.
*   **Серверная часть (Бэкенд) — Опционально:**
    *   Если требуется синхронизация между устройствами (телефон + ПК), сохранение истории и социальные функции, нужен сервер с базой данных (БД).
    *   Если приложение строго локальное (в целях приватности), сервер не нужен — все данные хранятся на устройстве пользователя.

---

### 3. ОСНОВНОЙ ФУНКЦИОНАЛ ПО ПУНКТАМ

#### 3.1. Модуль сбора статистики
*   **Отслеживание в реальном времени:** Фиксация времени запуска и закрытия приложений/сайтов из списка соцсетей.
*   **Категоризация:** Автоматическое распределение отслеживаемых ресурсов по категориям (Видео, Мессенджеры, Соцсети, Развлечения).
*   **Панель статистики:**
    *   Сегодняшняя активность (общее время, время в каждом приложении).
    *   История за неделю/месяц в виде графиков.
    *   Рейтинг самых посещаемых ресурсов за период.

#### 3.2. Модуль установки лимитов
*   **Дневные лимиты:** Установка временного ограничения (например, 30 минут) на конкретное приложение или категорию.
*   **Временные окна (Расписание):** Возможность задать часы, в которые соцсети доступны (например, только в обед с 13:00 до 14:00).
*   **Лимиты по часам:** Ограничение использования, например, не более 5 минут каждого часа.
*   **Напоминания:** Предупреждение за 5 минут до окончания лимита.

#### 3.3. Модуль блокировки
*   **Мягкая блокировка:** Уведомление о превышении лимита с возможностью его проигнорировать (отложить блокировку на 5 минут).
*   **Жесткая блокировка:** Приложение/сайт становится недоступным. При попытке открыть соцсеть пользователь видит кастомную заглушку.
*   **Режим "Надзор":** Возможность подключить "куратора" (друга), которому придет отчет о срыве блокировки, или который может удаленно заблокировать устройство.

#### 3.4. Мотивационный модуль (Геймификация)
*   **Система достижений:** Награды за соблюдение лимитов ("3 дня без срывов", "Не заходил в TikTok неделю").
*   **Визуальный прогресс:** "Саженец фокуса", который растет, пока вы не отвлекаетесь.
*   **Челленджи:** Встроенные испытания, например, "Не использовать телефон 2 часа".

---

### 4. ТРЕБОВАНИЯ К РЕАЛИЗАЦИИ

#### 4.1. Функциональные требования
*   Приложение должно корректно определять, находится ли целевое приложение на переднем плане (активно).
*   Блокировка должна срабатывать с точностью до секунды при достижении лимита.
*   Настройки должны сохраняться и применяться мгновенно без перезагрузки устройства.
*   Статистика должна обновляться в реальном времени на дашборде.

#### 4.2. Нефункциональные требования
*   **Приватность и безопасность:**
    *   Сбор только тех данных, которые необходимы для работы (время использования, названия приложений).
    *   Отсутствие сбора контента (переписок, фото, лайков).
    *   Четкое информирование пользователя о том, какие данные собираются и куда они передаются (или что хранятся только локально).
*   **Производительность:**
    *   Трекер не должен расходовать батарею устройства больше, чем на 2-3% в день.
    *   Приложение не должно тормозить работу смартфона или браузера.
*   **Стабильность:**
    *   Блокировка не должна сниматься при перезагрузке телефона (если пользователь не отключил её намеренно).
    *   Приложение должно иметь защиту от "детских" способов обхода (удаление приложения, перевод времени на телефоне).
*   **UI/UX (Дизайн):**
    *   Интуитивно понятный интерфейс настройки лимитов (не более 2-х кликов до установки лимита).
    *   Приятный, ненапрягающий дизайн экрана блокировки (не агрессивный, а мотивирующий).
    *   Поддержка темной и светлой темы.
